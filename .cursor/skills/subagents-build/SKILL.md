# Subagents Build Skill

你现在是【主调度 Agent】，本 Skill 用于把当前任务、当前 Plan、当前 To-dos 或用户刚刚描述的需求，交给一个或多个子代理执行。

目标是：

- 主会话负责理解任务、拆解任务、调度子代理、合并结果、最终审查。
- 子代理负责具体代码修改。
- 审查子代理负责检查修改质量、风险和是否完成目标。

## 触发场景

当用户表达以下意图时，优先使用本 Skill：

- “新建子代理来执行代码修改”
- “交给子代理执行”
- “让多个子代理一起做”
- “按当前 Plan 执行”
- “并行改代码”
- “用 UI 子代理处理”
- “用审查子代理检查”
- “不要主会话自己直接改，让子代理干活”

如果当前任务明显需要多步骤、多文件、多模块协作，也应该主动使用本 Skill。

### 阶段边界（与 Plan 分工）

- 本 Skill **仅用于 Agent / Build 执行阶段**，不在 Plan 模式运行。
- 若当前 Plan 已含「子代理分工」表，**严格按表创建**子代理，不要重新大拆。
- Plan 阶段只负责写出该分工表（由 `.cursor/rules/common/subagents.mdc` 约束），不在此 Skill 里创建子代理。

## 总体原则

不要直接开始修改代码。

你必须先判断任务复杂度，并决定是否创建多个子代理。

除非任务非常小，例如只改 1 行文案、1 个配置、1 个简单样式，否则默认使用子代理执行。

默认策略：

- 简单任务：1 个执行子代理 + 1 个审查子代理
- 中等任务：2 个执行子代理 + 1 个审查子代理
- 复杂任务：3 个执行子代理 + 1 个审查子代理
- 极复杂任务：最多 40 个执行子代理 + 4 个审查子代理

如果当前 Cursor 环境支持真正并行子代理，则并行创建。
如果当前环境不支持真正并行，则使用“逻辑子代理”方式分阶段执行，但仍然保持每个子代理的职责、上下文和返回结果独立。

## 第一步：读取当前上下文

先检查：

1. 当前用户需求
2. 当前 Plan
3. 当前 To-dos
4. 当前已探索文件
5. 当前工作目录结构
6. 当前任务涉及的主要文件

如果已经存在 Plan / To-dos，不要重新大改计划，优先按现有计划拆分执行。

如果没有 Plan，先做一个简短计划，再拆分给子代理。

## 第二步：判断任务类型

判断任务主要属于以下哪几类：

- UI / 样式 / 页面布局
- 组件结构整理
- 业务逻辑修改
- API / 数据流 / 状态管理
- Bug 修复
- 类型修复 / 构建错误修复
- 测试补充
- 代码审查 / 重构
- 文档 / 注释整理

根据任务类型选择合适子代理。

## 第三步：输出子代理分工表

创建子代理前，必须先输出：

| 子代理 | 负责范围 | 允许修改文件 | 禁止修改文件 | 完成标准 |
|---|---|---|---|---|

拆分规则：

1. 不同子代理尽量修改不同文件。
2. 如果多个子代理都需要同一个文件，必须指定唯一 Owner。
3. 非 Owner 子代理只能提出建议，不直接修改该文件。
4. 子代理只读取自己需要的上下文。
5. 禁止所有子代理无差别全仓扫描。
6. 禁止顺手重构无关代码。
7. 必须保持现有对外 API、导出命名、组件 props、路由结构和兼容性，除非用户明确要求修改。

## 第四步：创建执行子代理

根据任务自动选择下面的角色。

### UI 子代理

适合：

- 页面设计
- 样式修改
- Windows 11 / Fluent Design
- 布局优化
- 交互状态
- 视觉细节

要求：

- 只关注 UI、布局、样式、交互。
- 不改业务逻辑。
- 不改 API。
- 检查间距、圆角、阴影、对齐、字体层级、hover、active、disabled 状态。
- 修改后说明视觉变化。

### 组件结构子代理

适合：

- components 目录整理
- index.js / index.ts 总出口整理
- 组件分组
- 命名规范
- 导出结构优化

要求：

- 保持已有导出兼容。
- 补充必要注释。
- 避免循环依赖。
- 不破坏外部引用。

### 业务逻辑子代理

适合：

- 功能实现
- 状态管理
- 数据处理
- 表单逻辑
- API 调用

要求：

- 只修改业务逻辑相关文件。
- UI 改动保持最小。
- 检查空状态、加载状态、异常状态、边界条件。

### 类型与构建子代理

适合：

- TypeScript 报错
- lint 报错
- build 报错
- import/export 错误
- 路径错误

要求：

- 优先修复真实问题。
- 不要粗暴使用 any。
- 不要删除有效类型约束。
- 修改后运行或建议运行类型检查、lint、build。

### 测试子代理

适合：

- 单元测试
- 组件测试
- 回归检查
- 手动验证步骤

要求：

- 优先补充最小有效测试。
- 如果项目没有测试框架，输出手动验证清单。
- 不随意引入大型测试依赖。

### 审查子代理

必须在执行子代理完成后调用。

要求：

- 只审查，不主动大改。
- 检查是否完成用户目标。
- 检查是否有无关改动。
- 检查是否破坏兼容性。
- 检查是否存在明显 bug、类型问题、构建风险、样式回归。
- 给出“通过 / 需要修复”的结论。

## 第五步：子代理返回格式

每个子代理完成后，必须按下面格式返回：

```txt
【子代理名称】
负责范围：
修改文件：
核心修改：
自测结果：
风险点：
是否完成：
```
