# Plan 结束后的 UI 实施分工

当主会话处于 plan 模式时，主会话负责：
- 调研现状
- 收敛方案
- 明确模块边界、数据流和影响面
- 输出可执行计划

当 plan 已结束、用户已确认进入实施，并且任务属于本仓库的 UI / SciterJS 前端修改范围时，主会话不要直接改代码，应优先调用 `/ui` 子代理执行实施。

## `/ui` 子代理定位

- `/ui` 子代理的定义文件位置是 `.cursor/agents/ui.md`。
- 主会话提到“调用 `/ui` 子代理”时，指的就是使用这个 `ui` agent，而不是自行猜测其他位置。
- 如果主会话需要确认 `/ui` 的职责、边界或提示词来源，应优先读取 `.cursor/agents/ui.md`，不要只依赖规则里的简称。

UI / SciterJS 前端范围包括但不限于：
- `src/layout/`
- `src/views/`
- `src/components/`
- `src/dialogs/`
- `src/styles/`
- 以及与界面呈现直接相关的 `src/assets/`、`src/data/`

分工约束如下。

## 主会话职责

- 在 plan 阶段完成调研、边界划分与方案确定。
- 在实施前判断任务是否属于 UI / SciterJS 前端范围。
- 若属于上述范围，把代码修改工作交给 `/ui` 子代理执行。
- 触发 `/ui` 前，主会话应明确以 `.cursor/agents/ui.md` 作为子代理定位与职责依据。
- 在 `/ui` 完成后，由主会话负责结果验收、风险说明，以及与非 UI 范围改动的协调。

## /ui 子代理职责

- 先读取 `AGENTS.md`。
- 先读取 `.agents/rules/DoNot.md`。
- 按任务类型继续读取相关 UI 规则。
- 在现有 SciterJS + Reactor + ES Modules 约束下，以最小 diff 完成实现。
- 优先复用现有布局、styleset、组件、图标与窗口模式，不自行扩散抽象。

## 例外

以下情况不强制使用 `/ui` 子代理，可由主会话或其他更合适的子代理处理：
- 非 UI 范围任务，例如 git、脚本、构建、配置、文档、后端、数据修复。
- 纯计划讨论、纯只读排查、纯规则整理。
- 明显不需要进入代码实施阶段的回答型任务。

## 执行顺序

主会话(plan)
-> 调研 / 定边界 / 出计划
-> 用户确认
-> 主会话先以 `.cursor/agents/ui.md` 确认 `/ui` 子代理位置与职责
-> `/ui` 子代理实施 UI 代码修改
-> 主会话验收与收尾
